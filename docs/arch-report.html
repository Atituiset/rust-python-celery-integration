千万级代码库异步扫描架构分析报告

架构范式: 跨语言异构微服务 (Rust + Python + Redis) | 协议标准: Celery Protocol v2/v5

1. 架构范式与核心价值

本架构采用 "Rust (网关/调度) + Redis (总线/状态) + Python (业务逻辑/AI算力)" 的异构组合，完美解决了单一语言在大型工程中的瓶颈：

  • Rust 端 (code-inspector-main)：承担极高吞吐量的代码库接入、AST初步扫描与任务拆分。其内存安全与低开销确保调度层面对海量并发时不产生OOM或GC停顿。
  • Python 端 (code-context-service)：专注于安全编译选项扫描（secure_compilation_ops_scan）及基于大模型的上下文解析。利用Python丰富的AI生态降低业务逻辑迭代成本。
  • 物理隔离与解耦：通过 Celery 协议约定，将 Broker（消息分发，DB 6）与 Backend（结果存储，DB 7）物理分离。两端服务互不阻塞，单点宕机不影响整体接入层。

4. API 约束闭环与执行时序

按照架构图流向，跨语言交互需严格遵守以下接口契约：

第一步 (L1)：Rust 写入队列
Rust 端将任务序列化为标准 Celery JSON 格式，LPUSH 进入 Redis DB 6。约束条件：任务名称必须与 Python 端的 name="secure compilation options scan" 完全一致，且忽略 Python 端的 self 参数绑定。
第二步 (L2)：Python 阻塞消费
Python Celery Worker 采用 BRPOP 监听 DB 6。读取到 payload 后，自动路由至本地对应函数 secure_compilation_ops_scan_work 并执行扫描逻辑。
第三步 (L3)：Python 回写状态
任务结束时返回 Dict 数据（如 {"status": "ok"}），Celery 框架拦截返回值，将其自动包装为包含状态码的 JSON，并存入 Redis DB 7。
第四步 (L4)：Rust 异步提取
Rust 轮询协程连接 DB 7 查询 celery-task-meta-<task_id>。一旦检测到状态为 SUCCESS，即提取真实数据，完成流水线的最终闭环。

5. 生产环境高可用建议

  • Worker 防丢机制：在 Python 端启动 acks_late=True。若 Worker 节点在计算上下文时遭遇 OOM 被系统强制关闭，任务消息将自动退回 DB 6，由其他节点重新消费。
  • 轮询性能优化：建议摒弃 Rust 侧高频轮询 DB 7（GET操作）。通过开启 Redis 的 Keyspace Notifications（键空间通知），Rust 可通过 Pub/Sub 模型订阅 DB 7 的更新事件，将毫秒级延迟与零无效网络IO完美结合。
